## 前提：阅读项目规范文档

**目标**：阅读前后端 docs 中的项目规范文件，理解当前项目的架构、分层、依赖方向和编码规范，为后续步骤建立规范认知。

**执行方式**：
- 系统阅读后端 `docs/` 目录下的所有规范文件（type.rules.md、boundary-contract.rules.md、entity.rules.md、modules.rules.md、usecase.rules.md、queryservice.rules.md、capability.rules.md、capability-plugin.rules.md、core.rules.md、infrastructure.rules.md、adapters.rules.md、worker-adapter.rules.md 等）
- 系统阅读前端 `docs/` 目录下的所有规范文件（layer-model.md、dependency-rules.md、stable-clean/architecture.md、infrastructure-rules.md 等）
- 理解后端分层 `adapters → usecases → modules → infrastructure` + `core` + `types` 的依赖方向
- 理解前端三维结构 `stable / labs / sandbox` × `app / pages / widgets / features / entities / shared` 的依赖规则

**预期成果**：建立完整的规范认知，确保后续步骤中的判断有据可依。

---

## 第一步：执行升级计划

**文件**：`plans/update/framework-update-upgrade-prompt.md`

**目标**：将老师最新版本框架的改动合并进当前项目，同时完整保留所有业务代码和定制功能。

**预期成果**：
- 纯框架文件对齐到老师最新版
- 混合修改文件完成智能融合（框架升级采纳 + 业务代码保留）
- 新增文件复制到位、过时框架文件清理
- 依赖与配置合并完成
- 项目可正常编译启动

---

## 第二步：执行检查计划

**文件**：`plans/update/framework-update-check-prompt.md`

**目标**：全面验证升级是否成功，确保老师最新版框架的所有改动已被正确应用，且业务定制完整保留。

**预期成果**：
- 纯框架文件对齐情况确认
- 业务文件融合情况确认（无框架更新遗漏、无业务代码丢失）
- 新增/删除文件情况确认
- 依赖与配置检查通过
- 无 Git 冲突标记残留
- 生成详细检查报告，列出需人工修复的项

---

## 第三步：执行修复计划

**文件**：`plans/update/framework-update-fix-prompt.md`

**目标**：对照项目 docs 中的规范，全面排查框架更新后可能引入的依赖方向违规、同功能二次实现、规范违反等问题并修复。

**预期成果**：
- 依赖方向违规全部消除或标注为已知偏差
- 同功能二次实现统一为单一真源
- 违反 docs 规范的代码修正完毕
- 常量值跨层一致性确认
- TypeScript 编译 + ESLint 检查通过
- 生成修复报告，列出已知偏差和待人工确认项

---

## 第四步：功能回归验证

**目标**：全局检查项目中的功能是否都能正常运行，排查是否因先前更改（升级/检查/修复）而引入了新的 bug。

**执行方式**：
- 启动前后端项目，验证所有关键功能可用
- 通过 GraphQL Playground / API 请求测试核心业务流程（登录、注册、博客 CRUD、AIGC 功能等）
- 检查控制台/终端是否有运行时报错
- 重点验证先前修复涉及的模块（如 BullMQ 常量迁移、Capability 模式迁移、模块导入变更等）
- 若发现 bug，立即修复并重新验证

**预期成果**：
- 所有业务功能正常运行
- 无因先前更改导致的新 bug
- 前后端启动无报错，核心 API 可正常响应

---

## 第五步：规范最终检查

**目标**：再次阅读 docs 中的项目规范文件，全局检查项目是否存在同功能二次实现和违反项目规范的地方，作为最终兜底检查。

**执行方式**：
- 重新审视 docs 规范文档，对照实际代码逐项排查
- 重点检查：同功能二次实现、依赖方向违规、类型定义重复、常量值分叉、Entity 污染、QueryService 写操作等
- 对第三步修复后仍可能遗留的规范问题进行最终扫描

**预期成果**：
- 项目代码与 docs 规范完全对齐，或偏差项已明确标注
- 无二次实现残留
- 无隐性规范违规

---

## 第六步：启动前后端项目

**目标**：启动前后端项目，验证项目可正常运行。

**端口要求（强制）**：
- **前端端口：16100**
- **后端端口：16200**

**执行方式**：
- 启动后端：在 `aigc-friendly-backend` 目录下执行启动命令，确保后端监听 16200 端口
- 启动前端：在 `aigc-friendly-frontend` 目录下执行启动命令，确保前端监听 16100 端口
- 确认两端均正常启动，无启动报错

**预期成果**：
- 前端在 `http://localhost:16100` 可访问
- 后端在 `http://localhost:16200` 可访问
- 无启动错误或警告

---

## 第七步：生成更新日志

**日志文件**：`plans/updatelog.md`

**目标**：阅读现有更新日志的格式，为本次更新生成同等格式的日志，插入到 `updatelog.md` 最下方。

**执行方式**：
- 阅读 `plans/updatelog.md` 中已有日志的格式结构
- 按相同格式生成本次更新的日志，包含：版本号、升级结果、本次修复、对齐确认、后续维护注意
- 将新日志插入到 `updatelog.md` 文件最下方

**日志格式模板**：
```markdown
## vX.X.X 框架升级更新日志
升级结果：[成功/部分成功]，XX.X% 框架文件与模板完全一致。

### 本次修复
- [修复项1]
- [修复项2]

### 对齐确认
- [确认项1]
- [确认项2]

### 后续维护注意
1. [注意项1]
2. [注意项2]
```

**预期成果**：
- `updatelog.md` 末尾新增一条格式一致的更新日志
- 日志内容准确反映本次更新实际情况

---

**目标**：将本文档开头"执行进度"表格中的所有"已完成"状态修改为"未完成"，以便 `updateplans.md` 可重复使用。

**执行方式**：
- 将执行进度表格中所有步骤的状态改为"未完成"
- 保留步骤名称和表格结构不变

**预期成果**：执行进度表格恢复为全部"未完成"状态，文件可再次用于下一轮框架更新。
