以下为“升级”提示词


【项目背景】
我正在开发一个 AIGC 网页项目，最初是基于老师的网站基础框架创建的。
现在老师的框架在 GitHub 上发布了新版本，我需要将我的项目升级到老师的最新版本，
同时完整保留我已经添加的所有业务代码和定制功能。

【当前状态】
- 我的项目根目录：当前打开的 Trae 工作区。
- 老师最新框架的完整源代码的绝对路径位于：~/aigc-template
- 我的项目里可能包含一些从原框架复制过来后又修改过的文件，以及我全新创建的文件。
- 两个目录之间没有共享的 Git 历史，请纯粹基于文件内容进行对比和合并。

【你的任务】
对比这两个目录，将老师最新版本的改动合并进我的项目，直接在本项目中修改文件。

【合并规则】
1. 保护我的业务代码：
   所有 AIGC 功能、自定义页面、业务逻辑、个性化配置（密钥、地址、开关、样式等）必须保留，不可丢失。
2. 采纳框架升级：
   框架的结构性升级、安全修复、性能优化、新增通用功能，优先采纳。
3. 文件处理策略（请严格遵循）：
   a) 如果某个文件在我的项目中存在，但老师最新版中不存在：
      - 如果该文件是我自己创建的（业务文件），保留。
      - 如果该文件是框架原有文件，但老师在新版中删除了，且我的业务不依赖它 → 可以删除（谨慎操作，不确定则保留并添加注释说明）。
   b) 如果某个文件在老师最新版中存在，但我的项目中没有：
      - 直接复制到我的项目对应位置。
   c) 如果两个目录中同名文件内容不同：
      你需要进行智能合并：
      - 对于纯框架文件（如工具库、通用组件），以老师新版为准。
      - 对于明确包含业务逻辑的文件（如页面组件、API 接口、AIGC 相关模块），融合双方修改：
         若老师改动为通用优化且不影响业务 → 用老师版本。
         若老师改动与我的业务逻辑冲突 → 保留我的版本，并在代码中用注释 // [KEPT:业务保留] 标明。
         若可融合 → 融合两者，并加注释 // [MERGED]。
      - 对于依赖文件（package.json / requirements.txt / pom.xml 等）：
         保留我新增的依赖；加入老师新增或升级的依赖；版本冲突时以老师为准，但保留我的依赖并注释说明用途。
      - 对于配置文件（.env / config.js / application.yml 等）：
         保留我的配置值；引入老师新增的配置项并设为默认值（可用注释说明）。
4. 最终结果需确保项目可以正常启动，所有业务功能不受影响。
5. 如果有任何不确定的地方，保留我的版本并在代码中添加注释说明，最后列出需要人工确认的项。

【执行方式】
- 请先全面扫描两个目录的差异，然后逐个文件执行合并。
- 操作完成后，给我一份简短的合并总结：哪些文件自动合并了，哪些进行了冲突融合，哪些完全保留了我的版本，哪些新增或删除。
- 如果有文件无法自动处理或需要我决定，请明确指出。

现在开始执行。







以下为”检查“提示词


【任务背景】
我之前使用老师的网站基础框架搭建了 AIGC 项目，现在刚完成一轮手动 + AI 辅助的框架升级，
目标是把项目对齐到老师在 GitHub 上发布的最新版本。
我需要你帮我全面检查这次升级是否成功。

【当前状态】
- 我的项目根目录：当前打开的 Trae 工作区。
- 老师最新框架的完整源代码的绝对路径位于：~/aigc-template
  这个目录就是用来对齐的目标版本，内容完全纯净，没有我的业务代码。

【你的任务】
仔细对比这两个目录，判断我的项目是否已经成功应用了老师 最新版本 框架的所有改动，
同时完整保留了我的业务定制。请逐项检查并生成一份详细的检查报告。

【检查规则】
1. 纯框架文件（如工具函数、通用组件、样式库、没有业务改动的模块）：
   - 检查我的项目中的版本是否与老师最新版完全一致。
   - 如果不一致，说明我可能漏掉了更新，或者错误地覆盖了框架文件。
   - 标注出所有存在差异的纯框架文件，并指出差异程度（完全缺失、部分不同等）。

2. 混合修改文件（即既有我的业务逻辑，又应有框架更新的文件）：
   - 检查我的文件是否同时包含了：
     a) 老师新增的框架功能（从旧版到新版增加的结构、方法、配置等）
     b) 我原有的业务逻辑（AIGC 功能、自定义页面、特殊配置等）
   - 如果发现某个文件里老师新增的关键功能缺失，标记为“框架更新遗漏”。
   - 如果发现某个文件里我的业务代码被误删或覆盖，标记为“业务代码丢失”。

3. 新增与删除文件：
   - 检查老师新版中新增的文件，我的项目中是否存在。
   - 检查老师新版中删除的框架旧文件，我的项目中是否已正确删除（如果该文件不涉及我的业务）。
   - 如果某个框架已删除的文件因为我业务需要而被保留，请确认并说明这是合理的。

4. 依赖与配置：
   - 对比 package.json / requirements.txt 等依赖文件：我是否引入了老师新增的依赖，同时保留了我自己的依赖。
   - 对比 .env / config 等配置文件：我是否引入了老师新增的配置项，同时保留了原有的自定义值。

5. 代码冲突残留：
   - 扫描所有文件，检查是否还存在 Git 冲突标记（<<<<<<<, =======, >>>>>>>），如果有，必须明确指出。

【报告格式要求】
请输出一份结构清晰的检查报告，包含以下部分：
1. 总体结论：升级是否整体成功，是否存在重大问题。
2. 纯框架文件对齐情况：列出所有仍需对齐的文件及差异说明。
3. 业务文件融合情况：指出哪些文件中发现框架更新遗漏或业务代码丢失。
4. 新增/删除文件情况：列出老师新增但缺失的文件，老师删除但未删除的文件，以及合理保留的删除文件。
5. 依赖与配置检查结果。
6. 冲突标记残留检查结果。
7. 建议：哪些地方需要我手动修复，或者是否需要再次执行局部合并。

请开始检查。
